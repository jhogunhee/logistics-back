1. OMS (주문)

주문 원장 영역. WMS 작업문서(입고예정·출고주문)의 발생지. 등록 폼과 목록·취소 관리 화면을 분리한다.

화면주요 기능상태API
입고주문 벤더 발주 등록 폼. 벤더/입고예정일/라인(SKU·수량) 입력 미구현 신규 — 주문 엔티티 필요
/oms/inbound-order
입고주문 관리 목록 검색(주문번호/벤더/상태/기간), 수정·취소, 상태별 진행. WMS 입고예정(ASN) 생성의 기점 미구현 신규
/oms/inbound-orders
출고주문 점포 수주 등록 폼. 점포/납품일/라인(SKU·수량) 입력 구현 POST /outbound/orders, /{id}/lines
/oms/outbound-order
출고주문 관리 목록 검색 + 취소, 상태별 진행 표시. (기존 "출고주문 관리" 화면을 OMS로 이관) 구현 GET /outbound/orders, /{id}/cancel
/oms/outbound-orders

미결정: OMS 입고주문과 WMS 입고예정(ASN)의 관계 — ① 주문 확정 시 ASN 자동 생성(ASN 화면은 조회·검수 전용이 되고 등록 기능 제거) ② ASN을 계속 독립 등록(주문 없이도 입고 가능) 중 택일 필요. 출고는 주문이 곧 WMS 작업 대상이라 이 문제가 없음.

2. 입고 (Inbound)

화면주요 기능상태API
입고예정(ASN) 관리 목록 검색(입고번호/벤더/상태/기간), 등록, 취소. 헤더–라인 2단 구조 구현 GET/POST /inbound/asns
/{id}/lines, /{id}/cancel
입고 검수 라인별 합격/불합격 수량 + 제조일자 입력 → Lot 채번 후 RCV-STAGE 입고. 하단에 검수 이력 + 건별 취소 구현 POST /{id}/receive
GET /{id}/lines/{lineId}/receipts
POST /{id}/receipts/{invHistId}/cancel
적치지시 스테이징 잔량이 있는 (라인, Lot) 배치에 대해 적치 전략에 따라 로케이션을 배정한 지시를 생성. 지시 목록 조회 / 재배정 / 취소 미구현 신규 — 지시 엔티티 + 전략 엔진 필요
적치 발행된 적치지시를 선택해 실행(보관 로케이션으로 MOVE) → 지시 소진. 지시 로케이션과 다른 곳에 넣은 경우 실적 로케이션 수정 허용 여부는 결정 필요 진행중 GET /inbound/putaway/lines
/lines/{id}/candidate-locs
POST /lines/{id}
→ 지시 기반으로 재작성
입고확정 입고 흐름의 마지막 단계. 라인별 예정/검수/적치/결품 수량을 보여주고, 결품(expctQty − rcvdQty) 사유 입력 후 확정 → RECEIVING → COMPLETED.
선행 조건: 전 라인 미적치 잔량 0 — 미적치가 남으면 확정 버튼 비활성 + 사유 표시(검수만 하고 확정하는 것은 업무적으로 성립하지 않음)		 미구현 POST /{id}/close 존재하나 의미가 달라 재작성 필요 (현재는 RECEIVING→RECEIVED 중간 전이)

3. 재고 (Inventory)

화면주요 기능상태API
현재고 조회 SKU + 로케이션 + Lot 단위 스냅샷. 가용재고(onHand − alloc) 파생 컬럼, 유통기한 임박 강조 구현 GET /inventory/stock
재고 이력 조회 RECEIVE / MOVE / ADJUST / PICK / SHIP 원장. 참조문서번호로 입·출고 역추적 구현 GET /inventory/history
재고 속성변경 Lot 유통기한/제조일자 정정 등 수량이 아닌 속성의 수정. 변경 전/후 값을 이력에 남겨 추적. (재고상태 전환은 별도 재고 보류 화면에서) 미구현 신규
재고 보류 특정 재고(SKU+로케이션+Lot)를 보류 상태로 전환해 할당 대상에서 제외. 보류/보류해제, 사유 입력, 보류 재고 목록 조회. 유통기한 만료 Lot 자동 보류(배치) 여부는 추후 결정 미구현 신규 — inv에 재고상태 컬럼 추가 + 할당 로직에서 보류 재고 제외
재고조사 (실사) 조사 대상 지정(로케이션/존/SKU 범위) → 실사 수량 입력 → 전산 대비 차이 확인 → 확정 시 ADJUST로 스냅샷 보정 미구현 신규 — 조사 헤더/라인 엔티티 필요
재고 이동 보관 ↔ 보관 수동 MOVE (온도대 제약 검증 포함) 보류 우선순위 낮음

4. 출고 (Outbound)

화면주요 기능상태API
웨이브 편성 좌: 미편성 주문 / 우: 웨이브 소속 주문. 담기·빼기·해체.
웨이브 = 피킹지시 발행 단위 — 여러 주문을 묶어 한 번에 집품 지시를 내기 위한 그룹		 구현 GET/POST /outbound/waves
/{id}, /{id}/orders
DELETE /{id}/orders/{orderId}, DELETE /{id}
할당 출고주문 선택 → 라인별 주문수량 / 할당수량 / 잔량 표시. 자동할당(FEFO) 버튼과 수동할당(Lot·로케이션 선택 팝업), 할당해제. 점포 잔여수명 필터는 자동할당에 적용 미구현 신규 — v1의 유일한 할당 진입점
피킹지시 웨이브 단위로 일괄 발행. 웨이브 소속 주문들의 할당 레코드를 모아 로케이션 순으로 정렬한 지시를 생성. 지시 목록 조회 / 취소. 할당이 안 된 주문이 섞여 있을 때의 처리(제외 / 발행 차단)는 결정 필요 미구현 신규 — 지시 엔티티 필요 (적치지시와 대칭)
피킹 발행된 피킹지시를 보고 집품 → 수량 입력 → 보관 → SHIP-STAGE MOVE(tx_type=PICK). 지시 소진. 전 할당의 picked_qty == alloc_qty가 되면 주문 PICKED 전이 미구현 신규 — 지시 기반
출고확정 (Shipping) PICKED 주문의 SHIP-STAGE 반출 확정 → SHIPPED 미구현 —

추천 진행 순서

  1. OMS ↔ WMS 경계 결정 — 입고주문에서 ASN을 자동 생성할지, ASN 독립 등록을 유지할지. 이 결정에 따라 기존 ASN 등록 화면의 운명이 갈리므로 입고 쪽 손대기 전에 먼저 정한다. OMS 주문 엔티티 구현 자체는 뒤로 미뤄도 무방
  2. 입고 상태 모델 재정의 — 확정이 마지막 단계가 되면서 RECEIVED/COMPLETED 전이 지점이 바뀜. 코드 치기 전에 상태 기계부터 확정
  3. 적치 전략관리 → 적치지시 → 적치 — 전략이 없으면 지시를 만들 수 없으므로 이 순서 고정. 기존 적치 API는 지시 기반으로 재작성
  4. 입고확정 — 적치까지 붙은 뒤에야 결품 확정 화면이 의미를 가짐
  5. 할당(주문 단위) — v1의 유일한 할당 경로. 웨이브 릴리즈는 v1 제외
  6. 피킹지시(웨이브 단위) → 피킹 → 출고확정 — 출고 흐름 완성
  7. 재고 속성변경 / 재고조사
  8. 점포(Store) 관리 — 직접 구현
  9. 대시보드

기존 설계 결정변경 내용
"적치 지시 단계는 두지 않는다"지시 엔티티를 도입한다. 사유: 전략 기반 로케이션 배정을 하려면 배정 결과를 담을 레코드가 필요
"적치 로케이션 배정은 v1에서 수동 선택 / 자동 슬로팅·전략 엔진은 백로그"전략관리 + 자동 배정을 v1에 포함. 로케이션 용량 제약을 어디까지 볼지는 별도 결정 필요
"입고 마감 UI는 백로그, 버튼 미노출"입고확정으로 재정의하고 흐름의 마지막 단계로 이동. 검수 직후가 아니라 적치가 진행된 뒤 결품을 확정한다
"COMPLETED는 RECEIVED 상태 and 전 라인 putawayQty == rcvdQty 두 조건이 충족된 시점에 자동 전이"자동 전이 제거. 사용자의 명시적 확정 액션으로만 종료하며, 전 라인 적치 완료가 확정의 선행 조건이 된다
백로그(v1 제외): "폐기/홀드(재고 상태 + 만료 Lot 자동 홀드 배치)"재고 보류를 v1에 포함. inv에 재고상태를 추가하고 할당 후보에서 보류 재고를 제외한다. 만료 Lot 자동 보류 배치와 폐기는 계속 백로그
상태 RECEIVED제거. 검수/적치 진행도는 라인 수량에서 파생되므로 중간 상태가 불필요 — SCHEDULED → RECEIVING → COMPLETED
"할당은 오직 웨이브 릴리즈로만 일어난다"뒤집힘 — 할당은 주문 단위 화면에서만 일어난다. 웨이브 릴리즈는 v1 제외
"웨이브는 재고 할당의 단위다"웨이브는 피킹지시 발행 단위로 역할이 바뀐다. 할당은 주문 단위, 웨이브는 여러 주문의 집품을 한 번에 지시하는 그룹. 상태도 PLANNED/RELEASED가 아니라 "지시 발행 전/후"로 재정의 필요
주문 관리 화면은 WMS 입고/출고 메뉴 안에 있다OMS 그룹을 분리 신설. 입고주문·출고주문의 등록/관리를 OMS로 빼고, WMS 메뉴는 창고 작업(ASN·검수·적치·웨이브·피킹)만 남긴다. 기존 "출고주문 관리"는 /oms/outbound-orders로 이관
"피킹은 주문 단위 / 별도 지시 엔티티(pick_task)를 두지 않는다"뒤집힘 — 피킹지시 엔티티를 둔다. 웨이브 단위로 묶어 발행하려면 지시를 담을 레코드가 필요. 단, 집품은 지시 행(=할당 단위)대로 하고 SKU별 집약(배치 피킹)은 하지 않으므로 주문별 분류 공정은 여전히 불필요
